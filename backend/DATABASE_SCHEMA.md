# Coordina-AI Database Schema

## Overview

This schema supports:
- **Agent I/O**: All 5 agents' input/output structures
- **Frontend**: Tasks, teams, rubrics, risks, submissions
- **Memory Integration**: Activity tracking & decision logging synced from Redis
- **Persistence**: Long-term storage while Redis caches hot state

Core tables store normalized data; certain tables mirror memory folder structure for auditability.

---

## Core Tables

### 1. `projects`
Root project entity. Stores metadata & configuration.

```sql
CREATE TABLE projects (
  id VARCHAR(255) PRIMARY KEY,
  name VARCHAR(255) NOT NULL,
  description TEXT,
  
  -- Dates
  start_date DATE,
  deadline_date DATE,
  
  -- Status tracking
  status VARCHAR(50) DEFAULT 'created', -- created, in_progress, completed, archived
  workflow_stage VARCHAR(50) DEFAULT 'created', -- created, analysed, planned, coordinated, monitoring, validated
  
  -- Metadata
  team_size INT,
  confidence_score FLOAT, -- From instruction_analysis
  
  created_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP,
  updated_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP ON UPDATE CURRENT_TIMESTAMP
);
```

### 5. `tasks`
Tasks generated by planning_agent. Describes work breakdown.

```sql
CREATE TABLE tasks (
  id VARCHAR(255) PRIMARY KEY,
  project_id VARCHAR(255) NOT NULL,
  
  task_id VARCHAR(50), -- From agent output (e.g., "T1")

  description TEXT,
```

## Key Design Decisions

### 1. **Memory Folder Integration**
- `activity_events` syncs from `memory/activity_tracker.py` → DB for historical queries
- `decision_logs` syncs from `memory/decision_log.py` → DB for auditability
- Memory folder remains source-of-truth for real-time activity; DB stores archive

### 2. **JSON Fields**
Used for flexible, semi-structured data:
- `tasks.dependencies` — Task ID arrays
- `tasks.assigned_to` — Multiple assignees
- `team_members.skills` — Skill tags
- Agent full outputs (`full_output`, `output`) — Preserve exact GLM/agent responses

### 3. **Denormalization**
- `team_members.contribution_score` — Cached from `contribution_balance` for quick lookups
- `role_assignments.task_ids` — Avoids join on many-to-many
- Tradeoff: update consistency for query speed

### 4. **Timestamps**
- `event_timestamp` / `logged_at` — When event actually occurred
- `recorded_at` / `created_at` — When logged to DB
- Enables replay & audit trails

### 5. **Status Tracking**
- Task status: `pending`, `in_progress`, `done`, `backlog`
- Project workflow_stage: `created`, `analysed`, `planned`, `coordinated`, `monitoring`, `validated`
- Risk is_active: tracks resolution over time

## Query Patterns

### Frontend: Get Project Dashboard
```sql
SELECT p.*, 
  COUNT(DISTINCT t.id) as task_count,
  SUM(CASE WHEN t.status = 'done' THEN 1 ELSE 0 END) as completed_tasks,
  COUNT(DISTINCT tm.id) as team_size
FROM projects p
LEFT JOIN tasks t ON p.id = t.project_id
LEFT JOIN team_members tm ON p.id = tm.project_id
WHERE p.id = ?
GROUP BY p.id;
```

### Frontend: Get Active Risks
```sql
SELECT r.*, p.deadline_date
FROM detected_risks r
JOIN projects p ON r.project_id = p.id
WHERE r.project_id = ? AND r.is_active = TRUE
ORDER BY r.severity DESC, r.created_at DESC;
```

### Memory Sync: Activity in Last N Days
```sql
SELECT ae.* FROM activity_events ae
WHERE ae.project_id = ? AND ae.event_timestamp >= DATE_SUB(NOW(), INTERVAL 2 DAY)
ORDER BY ae.event_timestamp DESC;
```

### Audit: Agent Decision History
```sql
SELECT dl.* FROM decision_logs dl
WHERE dl.project_id = ? AND dl.agent = ?
ORDER BY dl.logged_at DESC
LIMIT 10;
```

## Indexes Summary

**High-traffic queries:**
- `projects.id` — PK
- `tasks.project_id, tasks.status` — Filter & list
- `team_members.project_id` — Roster lookup
- `activity_events.project_id, activity_events.member_id` — Activity search
- `detected_risks.project_id, detected_risks.is_active` — Active risk filtering
- `decision_logs.project_id, decision_logs.agent` — Agent audit trail

**Date-based queries:**
- `tasks.start_date, tasks.due_date` — Timeline/Gantt
- `milestones.due_date` — Milestone tracking
- `activity_events.event_timestamp` — Recent activity
- `risk_reports.executed_at` — Health snapshots

---

## TTL & Archival

- Keep active data: `projects`, `tasks`, `team_members` indefinitely
- Archive old data: Move `activity_events`, `decision_logs` after 90 days to archive table
- Risk/Submission reports: Keep for full project lifecycle + 1 year
